# Step 2: 계획

## 목적

분석 결과를 바탕으로 구현 계획을 수립합니다.
사용자 승인 후에만 개발 단계로 진행합니다.

## 선행 조건

- [ ] `01-analysis.md`가 태스크 폴더에 존재하고 완성됨
- [ ] `context-chain.md`를 읽고 Step 1의 결정사항을 확인함

> **위 조건이 충족되지 않으면 멈추고, 먼저 해결하세요.**

## 입력

- 분석 결과 (`01-analysis.md`)
- 컨텍스트 체인 (`context-chain.md`)
- 프로젝트 골든룰 (`docs/beliefs/golden-rules.md`)
- 개발 규칙 (`.harness/pipeline/rules.md`)

## 지침

### 1. 구현 전략 결정

- 기존 코드 패턴을 분석하여 일관된 방식 선택
- 참고문서(`docs/references/`)에서 해당 패턴 확인
- 신규 파일 생성 시 네이밍 규칙 명시

### 2. 파일별 변경 계획

- 파일별로 무엇을 어떻게 변경할지 구체적으로 기술
- 신규 파일은 템플릿/패턴 참조 명시
- 삭제할 파일이 있다면 사유 기록

### 3. DB 변경 계획 (해당 시)

- DDL 스크립트 작성 (태스크 폴더 `ddl/`에 저장)
- 롤백 DDL 포함
- **DB 변경은 사용자가 직접 실행** -- 선행 작업으로 분류

### 4. 수동 작업 분류

**선행 작업** (Step 3 전에 사용자가 완료해야 함):
- DDL 실행, 자동생성 파일 재생성, 뷰 수정 등

**후행 작업** (개발 완료 후 사용자가 수행):
- 버튼/메뉴 등록, 권한 설정, 데이터 입력 등

### 5. 롤백 계획 (고위험인 경우)

- 변경 파일 목록
- DB 변경 롤백 SQL
- 외부 시스템 영향
- 롤백 순서 및 검증 방법

## 산출물 템플릿

태스크 폴더에 `02-plan.md`로 저장:

```markdown
# 계획: {작업 설명}

> 날짜: {YYYY-MM-DD}
> 복잡도: {simple | complex | high-risk}
> Status: DRAFT

## 사용자 선행 작업 (Step 3 전 완료 필수)

- [ ] {선행 작업 1}
- [ ] {선행 작업 2}

## 구현 계획

### {파일 1 경로}
- 작업: {생성/수정}
- 변경 내용: {구체적 설명}

### {파일 2 경로}
- 작업: {생성/수정}
- 변경 내용: {구체적 설명}

## DB 변경 (해당 시)

DDL 스크립트: `ddl/` 폴더 참조

## 사용자 후행 작업 (개발 완료 후)

- [ ] {후행 작업 1}

## 롤백 계획 (고위험인 경우)

| 항목 | 내용 |
|------|------|
| 변경 파일 목록 | {파일 나열} |
| DB 롤백 SQL | {DDL/DML} |
| 롤백 순서 | {역순 나열} |
| 롤백 검증 | {확인 방법} |
```

## 사용자 승인

계획을 사용자에게 제시하고 응답을 기다립니다:
- **"승인"** -> `Status: DRAFT`를 `Status: APPROVED ({날짜})`로 변경 후 Step 3 진행
- **"수정 {피드백}"** -> 피드백 반영 후 재제시
- **"반려"** -> 파이프라인 중단, 사유 기록

## 자체 검증 (다음 단계 전 확인)

- [ ] Status가 APPROVED인가
- [ ] 선행 작업이 있다면 모두 `[x]`로 체크되었는가
- [ ] 신규 파일의 네이밍이 rules.md 규칙을 따르는가
- [ ] 고위험이면 롤백 계획이 포함되었는가

## 완료 후

1. `context-chain.md`에 핵심 결정사항 기록
2. 선행 작업 완료 확인 후 Step 3로 진행
