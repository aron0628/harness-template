# Step 6: 완료

## 목적

작업을 정리하고, 변경 사항을 요약하여 사용자에게 최종 승인을 받습니다.

## 선행 조건

- [ ] Step 5 통과 (`04-verify.md` 판정: PASS)
- [ ] `context-chain.md`를 읽고 전체 과정을 확인함

> **위 조건이 충족되지 않으면 멈추고, 먼저 해결하세요.**

## 지침

### 1. 변경 사항 요약 작성

- 전체 변경 내용을 간결하게 요약
- 변경 파일 최종 목록
- 주요 결정사항과 사유

### 2. 후행 작업 확인

- `02-plan.md`에 후행 작업이 있으면 사용자에게 안내
- 각 후행 작업의 수행 방법 가이드 포함

### 3. 품질 기록 갱신

- `docs/quality/drift-log.md`에 새로운 드리프트가 발견되었으면 기록
- `docs/quality/score.md`에 영향이 있으면 갱신

### 4. 태스크 폴더 정리

완료 승인 후:
- `docs/plans/active/{task-id}/`를 `docs/plans/completed/{task-id}/`로 이동

## 산출물 템플릿

태스크 폴더에 `05-summary.md`로 저장:

```markdown
# 완료: {작업 설명}

> 날짜: {YYYY-MM-DD}
> 복잡도: {simple | complex | high-risk}
> Status: PENDING APPROVAL

## 변경 요약

{전체 변경 내용을 1-3문장으로 요약}

## 변경 파일

| 파일 | 작업 | 설명 |
|------|------|------|
| {경로} | {생성/수정/삭제} | {변경 내용} |

## 주요 결정사항

- {결정 1: 사유}
- {결정 2: 사유}

## 사용자 후행 작업

- [ ] {후행 작업 1: 수행 방법}
- [ ] {후행 작업 2: 수행 방법}

## 사용자 테스트 시나리오

1. [ ] {테스트 항목 1}
2. [ ] {테스트 항목 2}
```

## 사용자 승인

변경 요약을 사용자에게 제시하고 응답을 기다립니다:
- **"승인"** -> `Status: APPROVED ({날짜})`로 변경, 태스크 폴더를 `completed/`로 이동
- **"수정 {피드백}"** -> 해당 Step으로 돌아가 수정

## 파이프라인 완료

모든 조건이 충족되면:
1. `05-summary.md`의 Status를 `APPROVED`로 변경
2. 태스크 폴더를 `docs/plans/completed/`로 이동
3. 파이프라인 종료
