# Step 1: 분석

## 목적

요구사항을 분석하여 범위, 영향 파일, 복잡도, 위험을 파악합니다.
계획 수립 전에 충분한 정보를 확보하는 단계입니다.

## 선행 조건

- [ ] 태스크 폴더가 `docs/plans/active/{task-id}/`에 생성됨
- [ ] `input/` 디렉토리에 분석 대상 파일이 준비됨
- [ ] `00-input.md`에 원본 요구사항이 기록됨
- [ ] `context-chain.md`가 초기화됨

## 입력 파일 준비

분석 대상 파일(기획서, 참고자료 등)을 태스크 폴더의 `input/` 디렉토리에 모읍니다.

### 파일 수집 규칙

| 상황 | 처리 |
|------|------|
| 다른 경로의 PDF | `input/`에 복사 |
| PPT/PPTX 파일 | PDF로 변환 후 `input/`에 저장 |
| 이미지 파일 | `input/`에 복사 |

### PPT -> PDF 변환

```bash
soffice --headless --convert-to pdf --outdir {task-folder}/input/ {원본.pptx}
```

> LibreOffice 필요. 설치: `brew install --cask libreoffice` (macOS) / `apt install libreoffice` (Linux)

### 특정 페이지 분석

PDF의 특정 페이지만 분석할 경우 `00-input.md`에 페이지 범위를 명시합니다:
```
분석 대상: input/기획서.pdf (12-15페이지)
```

## 입력

- 입력 파일 (`input/` 디렉토리)
- 원본 요구사항 (`00-input.md`)
- 프로젝트 아키텍처 (`ARCHITECTURE.md`)
- 개발 규칙 (`.harness/pipeline/rules.md`)

## 지침

### 0. 입력 파일 수집

- 사용자가 제공한 파일 경로를 확인
- PDF는 `input/`에 복사, PPT/PPTX는 변환 후 `input/`에 저장
- `00-input.md`에 수집된 파일 목록과 분석 대상 페이지 기록

### 1. 요구사항 추출

- `input/` 디렉토리의 기획서를 읽고 핵심 요구사항을 식별
- 필수(must-have)와 선택(nice-to-have) 구분
- 불명확한 부분이 있으면 사용자에게 질문

### 2. 아키텍처 매핑

- 이 작업이 어떤 도메인/모듈에 해당하는지 식별
- 화면 경로 파악 (UI 작업인 경우)
- 영향 받는 파일 목록 추정 (컨트롤러, 서비스, 뷰 등)

### 3. 영향 분석

- 코드베이스에서 관련 파일 검색 (grep/find)
- DB 테이블/뷰 변경 필요 여부 확인
- 외부 연동 관련 여부 확인
- 프레임워크 파일 인접 여부 확인 (rules.md FRM-* 규칙)
- **관련 데이터 관리 화면이 있으면 조회 패턴 분석**

### 4. 복잡도 판정

| 복잡도 | 기준 |
|--------|------|
| **단순** | 단일 파일 또는 경미한 변경, DB 변경 없음 |
| **복잡** | 다중 파일, DB 변경, 모듈 간 연동 |
| **고위험** | 금액/정산, 보안/인증, 외부 연동, 프레임워크 인접 |

### 5. 위험 식별

- 무엇이 잘못될 수 있는가?
- 사용자가 수동으로 해야 할 작업은 무엇인가?

```markdown
- [ ] {요구사항 2}

## 영향 파일

|------|------|------|

- 외부 연동: {예/아니오 -- 어떤 연동}
- 프레임워크 인접: {예/아니오 -- 어떤 파일}

- {위험 2}

```

## 자체 검증 (다음 단계 전 확인)

- [ ] 복잡도가 판정되었는가
- [ ] 영향 파일이 최소 1개 이상 식별되었는가
- [ ] `.harness/pipeline/rules.md`를 참조했는가
- [ ] DB 변경이 식별되면 복잡도가 최소 "complex"인가
- [ ] 외부 연동 관련이면 연동 포인트가 명시되었는가
