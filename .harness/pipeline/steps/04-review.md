# Step 4: 리뷰

## 목적

구현된 코드를 규칙과 계획에 대조하여 검토합니다.
CRITICAL/HIGH 이슈가 모두 해소될 때까지 수정-재리뷰를 반복합니다 (최대 3회).

## 선행 조건

- [ ] Step 3 완료 (코드 변경이 존재)
- [ ] `context-chain.md`를 읽고 이전 결정사항을 확인함

> **단순 복잡도이고 사용자가 리뷰 스킵을 요청한 경우 Step 5로 건너뛸 수 있습니다.**

## 입력

- 변경된 코드 (diff)
- 분석 결과 (`01-analysis.md`)
- 승인된 계획 (`02-plan.md`)
- 개발 규칙 (`.harness/pipeline/rules.md`)
- 프로젝트 골든룰 (`docs/beliefs/golden-rules.md`)

## 리뷰 체크리스트

### A. 규칙 준수

- [ ] 보안 규칙 (SEC-*) 위반 없음
- [ ] 네이밍 규칙 (NAM-*) 준수
- [ ] 코딩 패턴 (COD-*) 준수
- [ ] 프레임워크 보호 (FRM-*) 위반 없음
- [ ] DB 규칙 (DB-*) 준수

### B. 비즈니스 로직 정합성

- [ ] 요구사항(`01-analysis.md`)과 구현이 일치하는가
- [ ] 화면 흐름/버튼 동작이 올바르게 구현되었는가
- [ ] 계산 로직이 정확한가

### C. 기존 코드 패턴 일관성

- [ ] 동일 모듈의 기존 패턴과 일치하는가
- [ ] 새로운 패턴이 불필요하게 도입되지 않았는가

### D. 엣지케이스 및 안전성

- [ ] null/빈값 처리
- [ ] 배열/리스트 길이 불일치 처리
- [ ] 중복 데이터 방지
- [ ] 삭제 + 삽입 패턴에서 데이터 정합성

## 심각도별 조치

| 심각도 | 조치 | 리뷰 통과 |
|--------|------|----------|
| CRITICAL | 즉시 수정 필수 | 불가 |
| HIGH | 수정 필요 | 불가 |
| MEDIUM | 권고 사항으로 기록 | 가능 |

## 산출물 템플릿

태스크 폴더에 `03-review.md`로 저장:

```markdown
# 리뷰: {작업 설명}

> 날짜: {YYYY-MM-DD}
> 리뷰 라운드: {1/2/3}
> 판정: {PASS | FAIL}

## 발견 사항

| # | 심각도 | 규칙 ID | 파일:라인 | 설명 | 상태 |
|---|--------|---------|----------|------|------|
| 1 | {CRITICAL/HIGH/MEDIUM} | {SEC-01} | {파일:라인} | {설명} | {미수정/수정완료} |

## MEDIUM 권고 사항

- {기록할 권고 사항}

## 판정 근거

{PASS/FAIL 사유}
```

## 리뷰 프로세스

1. 리뷰 수행 후 결과를 `03-review.md`에 기록
2. CRITICAL/HIGH 이슈가 있으면:
   - 사용자에게 보고
   - 수정 방향 확인 후 코드 수정
   - 재리뷰 (최대 3회 반복)
3. 3회 후에도 미해소 -> 사용자에게 알림
4. 모든 CRITICAL/HIGH 해소 -> 판정: PASS

## 완료 후

1. `context-chain.md`에 리뷰 결과 요약 기록
2. Step 5로 진행
