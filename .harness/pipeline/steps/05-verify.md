# Step 5: 검증

## 목적

구현 결과가 계획과 일치하는지, 빌드/테스트가 통과하는지, 기존 기능에 영향이 없는지 검증합니다.

## 선행 조건

- [ ] Step 4 통과 (또는 단순 복잡도로 스킵됨)
- [ ] `context-chain.md`를 읽고 이전 결정사항을 확인함

> **위 조건이 충족되지 않으면 멈추고, 먼저 해결하세요.**

## 입력

- 변경된 코드
- 분석 결과 (`01-analysis.md`)
- 승인된 계획 (`02-plan.md`)
- `.harness/config.yaml`의 enforcement 명령어

## 검증 체크리스트

### 1단계: 빌드 확인

- [ ] 빌드 명령어 실행 (`config.yaml -> enforcement.build_command`)
- [ ] 신규 코드에서 발생한 에러/경고 확인
- [ ] PASS 기준: 빌드 성공, 신규 에러 0건

### 2단계: 테스트 확인 (설정된 경우)

- [ ] 테스트 명령어 실행 (`config.yaml -> enforcement.test_command`)
- [ ] 신규 테스트 실패 확인
- [ ] PASS 기준: 테스트 통과

### 3단계: 변경 파일 정합성

- [ ] `02-plan.md`의 변경 파일 목록과 실제 변경 파일 대조
- [ ] 누락된 파일 또는 계획에 없는 변경 여부 확인
- [ ] import 누락, 미사용 import 확인

### 4단계: 영향 범위 재검증

- [ ] `01-analysis.md`의 영향 분석과 실제 변경 범위 대조
- [ ] 수정한 파일의 기존 기능이 깨지지 않았는지 코드 레벨 확인
- [ ] 프레임워크 보호 파일 변경 여부 최종 확인

### 5단계: 사용자 테스트 시나리오 (안내용)

사용자가 직접 수행할 테스트 시나리오를 산출물에 포함합니다.

## 산출물 템플릿

태스크 폴더에 `04-verify.md`로 저장:

```markdown
# 검증: {작업 설명}

> 날짜: {YYYY-MM-DD}
> 판정: {PASS | FAIL}

## 자동 검증 결과

| 단계 | 결과 | 비고 |
|------|------|------|
| 빌드 | {PASS/FAIL} | {에러 내용} |
| 테스트 | {PASS/FAIL/N/A} | {실패 내용} |
| 파일 정합성 | {PASS/FAIL} | {누락/초과 파일} |
| 영향 범위 | {PASS/FAIL} | {위험 요소} |

## 변경 파일 최종 목록

| 파일 | 작업 | 상태 |
|------|------|------|
| {경로} | {생성/수정} | {확인됨} |

## 사용자 테스트 시나리오

1. [ ] {화면 정상 로딩 확인}
2. [ ] {데이터 표시 확인}
3. [ ] {핵심 기능 동작 확인}
4. [ ] {기존 기능 정상 동작 확인}
```

## 실패 시 조치

| 단계 | 실패 시 |
|------|---------|
| 빌드 | Step 3으로 돌아가 수정 |
| 테스트 | Step 3으로 돌아가 수정 |
| 파일 정합성 | 누락 파일 추가 또는 불필요 변경 원복 |
| 영향 범위 | 사용자에게 위험 요소 알림 |

## 완료 후

1. `context-chain.md`에 검증 결과 요약 기록
2. Step 6으로 진행
