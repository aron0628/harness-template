# Development Pipeline

> 기능 개발, 버그 수정, 다중 파일 변경 작업 시 이 파이프라인을 따르세요.
> 에이전트는 이 문서를 읽고 단계별로 진행합니다.

## 적용 대상

**사용해야 하는 경우:**
- 신규 기능 개발
- 다중 파일 수정이 필요한 버그 수정
- 새 모듈/화면 추가
- DB 스키마 변경

**사용하지 않아도 되는 경우:**
- 한 줄 수정으로 끝나는 단순 수정
- 문서만 변경
- 설정 파일만 변경

## 파이프라인 요약

| Step | 이름 | 산출물 | 게이트 | 자동 진행 |
|------|------|--------|--------|----------|
| 1 | 분석 | `01-analysis.md` | 복잡도 판정, 영향 파일 식별 | Yes |
| 2 | 계획 | `02-plan.md` | **사용자 승인 필수**, 선행 작업 완료 | **No** |
| 3 | 개발 | (코드 변경) | 빌드 통과 | Yes |
| 4 | 리뷰 | `03-review.md` | CRITICAL/HIGH 전부 해소 | Yes |
| 5 | 검증 | `04-verify.md` | 빌드 + 테스트 + 영향 범위 확인 | Yes |
| 6 | 완료 | `05-summary.md` | **사용자 승인** | **No** |

## 작업 시작하기

### 1. 태스크 폴더 생성

```
docs/plans/active/{YYYYMMDD}-{간단한-설명}/
```

### 2. 입력 문서 작성

태스크 폴더에 `00-input.md`를 생성하고 원본 요구사항을 기록합니다.

### 3. 컨텍스트 체인 초기화

`.harness/pipeline/context-chain.template.md`를 태스크 폴더에 `context-chain.md`로 복사합니다.

### 4. Step 1부터 순서대로 진행

`.harness/pipeline/steps/01-analyze.md`를 읽고 지침을 따릅니다.

## 게이트 조건 (단계를 건너뛰지 않는 방법)

다음 단계로 넘어가기 전 반드시 확인:
1. 이전 단계의 산출물 파일이 태스크 폴더에 존재하는가
2. `pipeline.yaml`의 해당 단계 게이트 조건이 충족되었는가
3. `auto_proceed: false`인 단계는 사용자가 명시적으로 승인했는가

**게이트 조건이 충족되지 않으면 멈추고, 먼저 해결한다.**

## 컨텍스트 체인

각 단계 완료 후 핵심 결정사항을 `context-chain.md`에 누적합니다.
다음 단계 시작 전 반드시 `context-chain.md`를 읽고 이전 결정을 확인합니다.

대화가 리셋되어도 컨텍스트가 유지됩니다.

## 사용자 체크포인트

두 단계에서 사용자 승인이 필요합니다:
1. **Step 2 (계획)** -- 계획을 제시하고 승인/수정/반려를 기다림
2. **Step 6 (완료)** -- 변경 사항 요약을 제시하고 승인을 기다림

계획에서 수동 선행 작업(DDL 실행, 버튼 등록 등)이 식별되면, Step 3 전에 완료되어야 합니다.

## 복잡도 라우팅

Step 1 분석 후 복잡도를 판정합니다:

| 복잡도 | 기준 | 계획 깊이 | 리뷰 |
|--------|------|----------|------|
| **단순** | 단일 파일, DB 변경 없음 | 간략 | 선택 |
| **복잡** | 다중 파일, DB 변경, 모듈 간 연동 | 상세 | 필수 |
| **고위험** | 금액/정산, 보안, 외부 연동, 프레임워크 인접 | 상세 + 롤백 계획 | 필수 |

상세 정의는 `pipeline.yaml`의 `complexity` 섹션을 참조하세요.
